<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Docker image → runtime | The image ships the yt-dlp/ffmpeg toolchain; stale binaries are an anti-bot/security exposure |
| render.yaml → deployed secrets | Secrets must never be committed; only sync:false references |
| SQS redrive → DLQ | Repeatedly failing messages must be quarantined (DLQ), not silently dropped |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06-01 | Information Disclosure | Secrets in render.yaml or Docker image | high | mitigate | All secrets are sync:false references in render.yaml (set in the Render dashboard), mirroring the existing onfeed-api pattern; no secret values committed; the setup script prints URLs for manual entry, never persists them |
| T-06-02 | Tampering | Stale/outdated ffmpeg or yt-dlp in the image | medium | mitigate | ffmpeg installed via apt (patched build, not the years-old @ffmpeg-installer bundle); yt-dlp pinned-current at build via explicit install; RESEARCH Open Question 2 notes a weekly self-update is a follow-up hardening, not phase-blocking |
| T-06-03 | Denial of Service | Failed messages silently dropped (no DLQ) | high | mitigate | setup-import-queue.sh provisions a DLQ with maxReceiveCount:3 redrive — closes the exact gap CONCERNS.md flags on the ingest queue; failed jobs are quarantined for investigation, not lost |
| T-06-04 | Elevation of Privilege | Worker container running as root | medium | mitigate | Runtime stage runs as non-root (USER node) mirroring the API Dockerfile, with a writable tmp dir for mkdtemp |
| T-06-SC | Tampering | apt/pip installs (python3, ffmpeg, yt-dlp) in the image | high | mitigate | Installed from official Debian apt + PyPI yt-dlp (yt-dlp is the de-facto downloader, RESEARCH-approved); pin/verify at build; no [SLOP]/[SUS] npm packages introduced here |
</threat_model>
