# Phase 3 — UI Design Contract

> Visual and interaction contract for `/import`, `/import/[jobId]` (progress), `/import/[jobId]/review`, and `/import/mine`. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — existing hand-rolled design system, no component library |
| Preset | not applicable |
| Component library | none (no shadcn, no Radix/base-ui) — all components are custom `web/components/*.tsx`, all icons are inline `<svg>` (no icon library installed) |
| Icon library | none — inline SVG paths, 24×24 viewBox, `strokeWidth 2`, `stroke="currentColor"`, matching `BottomNav.tsx`/`SearchForm.tsx` icon style |
| Font | Display: DM Serif Display (`--font-dm-serif`, `font-display` utility) for H1/H2 headings · Body/UI: Inter (`--font-inter`, default sans) — both loaded via `next/font/google` in `web/app/layout.tsx`, already global, do not re-load |

**Rationale for "Tool: none":** This project already has a mature, opinionated Tailwind v4 token system (`web/app/globals.css`) covering light+dark mode, semantic `--t-*` custom properties, a warm forest/terracota/creme palette, and ~20 existing hand-built components. Introducing shadcn now would fork the visual language mid-project and fight the existing `--t-*` token layer. Phase 3 MUST extend the existing system, not replace it. This is a deliberate, informed choice — not a skipped gate.

## Spacing Scale

Declared values (must be multiples of 4), matching the scale already in use across `web/app/(main)/*`:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps (`gap-1`), badge internal padding |
| sm | 8px | Compact element spacing (`gap-2`), chip padding |
| md | 16px | Default element/card spacing (`gap-4`, page `px-4`) |
| lg | 24px | Section-to-section spacing (`gap-6`/`gap-7`, matches `SearchForm`/`pantry` page pattern) |
| xl | 32px | Layout gaps between major page regions |
| 2xl | 48px | Reserved, not expected to be needed at screen sizes here |
| 3xl | 64px | Reserved, not expected to be needed at screen sizes here |

Exceptions: Progress screen stage indicator uses `gap-3` (12px) between stage rows — matches existing `flex flex-col gap-3` idiom used throughout `FavoritesList`/`SearchForm` for list-like stacks, not a new value (12 is not a declared token but is an established codebase micro-exception already present in `gap-3`/`py-3` usage sitewide — treat as inherited, not phase-specific).

## Typography

Matches the exact 4 sizes/2 weights already established across the codebase (`buscar`, `pantry`, `favorites` pages):

| Role | Size | Weight | Line Height | Font |
|------|------|--------|-------------|------|
| Body | 14px (`text-sm`) | 400 (regular) | 1.5 (`leading-relaxed`) | Inter |
| Label / meta | 12px (`text-xs`) / 10.5–11px (`text-[10.5px]`/`text-[11px]`) | 600 (semibold) | 1.4 | Inter |
| Heading (page H1) | 32px (`text-[2rem]`) | 700 (bold) | 1.2 (`leading-tight`) | DM Serif Display (`font-display`) |
| Heading (section/card title) | 20–24px (`text-xl`/`text-2xl`) | 600 (semibold) | 1.25 | DM Serif Display (`font-display`) |

Rules carried over from the existing codebase (not new):
- Page H1 always `font-display text-[2rem] font-bold leading-tight text-forest`, exactly as in `SearchForm.tsx`/`pantry/page.tsx`.
- Body copy under H1 always `text-sm text-carvao/55 leading-relaxed`.
- No font size outside this table is introduced by this phase. Two weights only: 400 (body) and 600/700 (semibold/bold for labels/headings) — no 500 (medium) introduced net-new.

## Color

The existing `--t-*` semantic tokens (light+dark, see `web/app/globals.css`) ARE the color system. Phase 3 introduces **zero new raw hex values** — every color below already exists as a token.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--t-bg-screen` (`#faf4e8` creme light / `#0e1812` dark) | Page background across all 4 new screens |
| Secondary (30%) | `--t-bg-card` / `bg-surface` (`#ffffff` light / `#162018` dark) + `border-areia` (`--t-bd-card`) | Cards: URL input card, stage-progress card, review-screen field cards, "Minhas importações" list rows |
| Accent (10%) | `bg-terracota` (`#d4644a`) | Reserved for exactly: (1) the primary CTA button "Colar link", (2) the primary CTA button "Iniciar importação" / submit, (3) the "Confirmar receita" button on the review screen. Never used for anything else on these screens (not borders, not icons, not secondary buttons). |
| Destructive | `--t-fat-bg` / `#c25a3c` (existing "over/danger" red-orange pairing already used in `AdaptButton.tsx`'s quota-exhausted tag) | "Descartar importação" / "Cancelar" destructive actions only |

Accent reserved for: **"Colar link" button, "Iniciar importação" submit button, "Confirmar receita" button on the review screen.** Nothing else on these 4 screens uses terracota — progress-stage active-state uses `forest` (see Grounding/Status Palette below), not terracota, to keep terracota meaning "primary forward action" exclusively.

### Grounding Badge Palette (the one new visual primitive this phase needs)

D-07 requires a badge distinguishing `grounded` (neutral, no highlight) from `inferred`/`ambiguous` ("confira isto"). This directly reuses the **existing `--t-warn-bg`/`--t-warn-fg`** token pair (already defined in `globals.css`, currently unused in any shipped component — reserved but not yet consumed) rather than inventing a new color:

| Grounding value | Visual treatment | Token |
|---|---|---|
| `grounded` | No badge shown at all. Field renders exactly like any normal editable field — neutral is the absence of a badge, not a "green OK" badge. (Do not add a checkmark/green badge for grounded — that would be a 4th color in a 2-color system and adds visual noise to fields that need no attention.) | none — default field styling |
| `inferred` | Small pill badge, label "Confira isto — inferido", amber/warn palette | `bg: var(--t-warn-bg)` (`#fff8e6` light / `#221a08` dark), `fg: var(--t-warn-fg)` (`#8a6200` light / `#cc9020` dark), `border: 1px solid color-mix(in srgb, var(--t-warn-fg) 25%, transparent)` |
| `ambiguous` | Same pill, label "Confira isto — impreciso" (distinguishes from "inferred" in copy only, not color — both need the same review action from the user) | same as `inferred` |

This is a deliberate 2-state visual system (attention / no-attention), not 3-color traffic-lighting — matches REV-02's actual requirement ("sinalizado para correção") without over-designing a distinction the user doesn't need to act on differently.

### Stage indicator (progress screen) visual spec
- Vertical list of 4 stages (queued → downloading → transcribing → extracting), each row: icon circle (32px) + label + status.
- Completed stage: `bg-forest text-creme` filled circle with a checkmark icon.
- Active stage: `border-forest` outlined circle with the existing `spin-ring` animation (reuse `LogoLoader.tsx`'s spin keyframe at small scale, not a new spinner).
- Pending stage: `bg-areia/30 text-carvao/40` circle, muted label.
- Connecting line between circles: 2px, `bg-forest` for completed segments, `bg-areia` for pending segments.

## Layout Notes

- All 4 screens live inside the existing `web/app/(main)/` layout shell (`TopBar` + `max-w-md` centered column + `BottomNav`) — no custom full-bleed layout. Matches every other secondary screen (`pantry`, `favorites`).
- `/import` entry point: added to `TopBar.tsx`'s `PAGE_TITLES` map as `"/import": "Importar receita"` (secondary-screen title bar, per D-01 "acessível pelo header/menu" — reachable via a link from `/perfil` or `/hoje`, exact anchor left to executor, but must NOT be a new `BottomNav` item — 5 slots are full and D-01 explicitly says "não é FAB global").
- `/import/[jobId]` (progress) and `/import/[jobId]/review`: also secondary screens, same shell. Add both path prefixes to `TopBar.tsx`'s `SELF_TITLED`/`PAGE_TITLES` handling as needed (progress screen can be self-titled with an in-body H1 like `pantry`/`SearchForm` do, to allow the stage-list to breathe under a big heading).
- `/import/mine`: same list-page shell as `/favorites` — H1 "Minhas importações" + `Link` back to `/import` in the header row (mirrors `FavoritesPage`'s header pattern with the "buscar" link).
- Mobile-first single column throughout (`max-w-md`), consistent with the entire existing app — no desktop-specific layout variant needed (codebase has none).
