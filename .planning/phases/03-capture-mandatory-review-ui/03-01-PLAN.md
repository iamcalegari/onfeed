<objective>
Add the backend confirm/edit endpoint and the "my imports" HTTP wrapper that Phase 3's frontend needs, plus the mongoat schema change that models a "confirmed" recipe. Today the API can enqueue (POST /import) and poll (GET /import/:jobId), but there is NO way to persist a user's inline edits, transition a recipe out of "awaiting review", or list a user's imports over HTTP — the review screen is unbuildable until these exist. This is genuine backend work inside a frontend-titled phase.

Purpose: Deliver REV-04's confirm gate (a recipe is only valid after an explicit successful PATCH), REV-03's edit persistence, and D-09's owner-scoped listing — all IDOR-safe by construction.

Output: `Recipe.confirmedAt` (BSON + TS + synced Atlas validator), `confirmImportedRecipe` service fn, `PATCH /import/:jobId/recipe`, `GET /import/mine`, `ImportRecipeEditSchema`, and Wave-0 Vitest coverage for the confirm gate + owner-scoping.

Scope note / known limitation (per resolved Open Question 1): editing an ingredient `name` persists the raw edited text but does NOT re-run `resolveCanonicalForIngestion` — this keeps the PATCH fast and synchronous (no Voyage call in the request path). A materially renamed ingredient may desync from its `canonicalId` until a later reconciliation pass. Name editing is NOT descoped — only re-canonicalization is deferred.
</objective>
