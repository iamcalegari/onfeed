<objective>
Close the enumeration/information-disclosure gap that Phase 3 creates: today `GET /recipes/:id` (recipe.routes.ts) has NO auth and NO ownership filter — it returns any recipe by id regardless of visibility. The moment unreviewed `private` + `reviewRequired` imports exist and are linkable by id (which this phase makes real), anyone who guesses/enumerates a recipeId could read a user's not-yet-reviewed, possibly-wrong extracted content before the owner has even seen it.

Purpose: Add a lightweight visibility guard so a private import is readable only by its owner, while keeping the route anonymously usable for public recipes.

Output: `getRecipeById` extended to resolve private-import ownership via `importJobId → ImportJob.userId`; the route wired to pass the caller's userId (soft-auth); a Vitest file proving owner-only access to a private import.

Critical grounding (verified against the codebase, NOT the naive assumption): imported recipes are persisted with `visibility:"private"` and `importJobId` set, but `createdBy[]` is NOT populated (see import.recipe-mapping.ts). The existing `getRecipeById(id, userId)` `$or` clause authorizes on `createdBy.userId` — which is EMPTY for imports — so reusing it as-is would 404 the legitimate owner. The guard MUST resolve ownership through `importJobId → ImportJob.userId`.
</objective>
