<objective>
Make every confirmed import immediately linkable via a secret token, and expose an unauthenticated public-by-token backend resolver that leaks no existence and never becomes an IDOR bypass for non-shared private recipes.

Purpose: SOC-01 (born private) is already satisfied at persist (visibility:"private"); this plan verifies it and adds the shareable-link half of SOC-02: a CSPRNG `shareSlug` minted in the SAME `confirmImportedRecipe` update that sets `confirmedAt` (D-04 — linkable immediately, no separate publish action), a pure `getRecipeByShareSlug(token)` lookup (D-03 — token is the sole door, resolve by token never by objectId), and a public `GET /recipes/share/:token` route (D-01 — unauthenticated read, uniform 404, like state when a session exists) that the frontend can use to redirect to the canonical `/recipe/[id]` once the recipe is public (D-12).
Output: shareSlug generation at confirm, `getRecipeByShareSlug`, the public share route, a repository/route test suite (token generation, no-existence-leak, IDOR-safety), and an updated recipes module README.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| anonymous internet -> GET /recipes/share/:token | The FIRST unauthenticated backend read surface; the token is the sole authorization, no session required |
| share token -> private recipe | A valid shareSlug is the only door to a private recipe's data; token entropy + uniqueness is the whole privacy model (D-03) |
| share route -> owner-only private guard | The public-by-token resolver must not become a bypass for the existing owner-only guard on NON-shared private recipes |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-08 | Information Disclosure (enumeration) | shareSlug token entropy | high | mitigate | CSPRNG (node:crypto randomBytes) >= 128 bits, URL-safe — brute-force/enumeration of the token space is computationally infeasible; resolve-by-token-never-by-objectId prevents id-sequence enumeration (D-03) |
| T-05-09 | Information Disclosure (existence leak) | GET /recipes/share/:token | high | mitigate | getRecipeByShareSlug returns null (never throws) and the route returns a uniform reply.notFound for missing/invalid/deleted tokens — a wrong token is indistinguishable from a nonexistent one |
| T-05-10 | Elevation of Privilege (IDOR bypass) | share route resolver | high | mitigate | The route resolves ONLY by shareSlug (never by objectId) and returns only a recipe whose token matches — a private, non-shared recipe has no shareSlug and is unreachable via this route; covered by test |
| T-05-11 | Spoofing (token re-mint / theft) | confirmImportedRecipe idempotency | medium | mitigate | shareSlug minted once, in the confirmedAt write; the existing confirmedAt idempotency guard prevents re-minting a new token on re-confirm |
| T-05-SC | Tampering | npm/pip/cargo installs | high | mitigate | Token generation uses node:crypto (stdlib) — no new dependency. If any install is proposed, gate via slopcheck + blocking human checkpoint for [ASSUMED]/[SUS] |
</threat_model>
