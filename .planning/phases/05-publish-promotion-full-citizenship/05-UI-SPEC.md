# Phase 5 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Brownfield note:** onFeed already has a mature, consistent design system (no shadcn, no Tailwind config file — Tailwind v4 CSS-first tokens in `web/app/globals.css`). This contract does **not** invent new visual language. It grounds four new/touched surfaces in what already exists and reuses two components verbatim (`ShareButton`, `LikeButton`).

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (no shadcn — hand-rolled component system, pre-existing) |
| Preset | not applicable |
| Component library | none (Radix/base-ui not used); native elements + inline `style` objects and Tailwind utility classes, mixed per existing convention |
| Icon library | none — inline SVG paths (see `LikeButton`, `ShareButton`, `BackButton`) and emoji glyphs (🎬 ⏱ 🍽 ✦ 📥) used as icons throughout |
| Font | Display: `var(--font-display)` → DM Serif Display (titles, hero numbers). Body/UI: `var(--font-sans)` → Inter (everything else) |

**components.json / Tailwind config:** absent. Colors are declared as CSS custom properties (`--color-forest`, `--color-terracota`, etc., in `@theme` block) consumed via Tailwind's v4 auto-generated utilities (`text-forest`, `bg-terracota/10`, `border-areia`) AND semantic `--t-*` tokens consumed via inline `style={{ background: "var(--t-bg-card)" }}`. New Phase 5 UI MUST use `--t-*` tokens for surfaces/borders/text (never hardcode hex) and Tailwind color names only for one-off accent utility classes (`text-terracota`, `border-terracota/40`) — matching the existing recipe detail page and `ResultCard`.

## Spacing Scale

Declared values (must be multiples of 4), matching existing recipe page/card spacing exactly:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, badge internal padding |
| sm | 8px | Compact stacks (badge row gaps, meta row gaps) |
| md | 16px | Default card padding, section gaps |
| lg | 24px | Section-to-section spacing (mirrors `marginTop: 20/24` on recipe page) |
| xl | 32px | — (not used at this surface's scale; reserved) |
| 2xl | 48px | Conversion CTA block breathing room on public share page |
| 3xl | 64px | Page bottom padding on public share page (matches `paddingBottom: 80` convention, rounded to scale — use 64 not 80 for new surfaces) |

Exceptions:
- Recipe hero image height stays **280px** fixed (non-8pt, inherited from `recipe/[id]/page.tsx` — do not change for the public share page; it must look identical).
- Touch targets for icon-only buttons (like, share) stay **38–44px** circular/pill, matching existing `LikeButton`/`ShareButton`/`BackButton` sizing.

- 10–11px / weight 600–800 — pill/badge text (`GroundingBadge`, `BadgePill`, rank badges)

Existing 60/30/10 split — Phase 5 introduces **zero new colors**. Reuse only.

Secondary semantic color (warn, not destructive) already exists and is reused, not invented:

## Surface-by-Surface Contract

### 1. Public share page — `/r/[token]` (SOC-01/02, D-01..D-04, D-12)

**Route placement:** New route OUTSIDE the `(main)` Clerk-protected group (e.g. `web/app/r/[token]/page.tsx`), sibling to `(main)`, so it renders without any auth gate. `web/middleware.ts` currently wraps `clerkMiddleware()` around effectively all non-static paths (matcher excludes only static assets) — this phase must add `/r/:path*` as a public route via Clerk's `createRouteMatcher`/`isPublicRoute` pattern (or confirm Next route-group placement alone is sufficient; verify at planning time, this is a backend/routing concern, not a visual one, but blocks the visual contract from rendering at all if misconfigured).

**Layout:** Byte-for-byte reuse of `recipe/[id]/page.tsx`'s visual structure — same 280px hero, same fade, same title/meta/intro block, same nutrition card, same ingredients list, same steps list, same `sourceMeta` credits block. Do NOT redesign; this is a read-only rendering mode of the same component tree, not a new page design. Concretely:
- Extract or share the JSX between `RecipePage` and a new read-only variant, OR pass an `isPublicVisitor: boolean` prop through — implementation detail for planner, but the VISUAL result must be pixel-identical to the authed recipe page for the parts that render.
- **Top action row** (currently `LikeButton` + `FavoriteButton` + `ShareButton` at hero top-right, `top: 60, right: 18`): keep `LikeButton` (with `canLike={false}` when logged out — existing prop already supports this exact state, see Interaction States below) and `ShareButton`; DROP `FavoriteButton` entirely for logged-out visitors (favoriting requires an account and has no logged-out affordance in this contract — do not add sign-in-redirect behavior to Favorite, it's out of scope; only Like gets the redirect treatment per D-01).
- **Account-only action CTAs inline in the page body** (`AdaptButton`, `AddToPlanButton`/`LogMealButton`, "Irei fazer esta receita" cook-mode CTA): render them exactly as they appear today, but wrap onClick/href with a sign-in redirect when `userId` is null, instead of conditionally hiding them (`{userId && ...}` becomes always-rendered, gated on click). This is the D-02 "visible but route to sign-in" requirement — do NOT grey them out or add a disabled visual state; they must look fully live to maximize conversion intent, per D-02's stated rationale.
- **Conversion CTA block** — NEW, appended after the steps list (bottom of page, before existing bottom padding): two-button stack, spacing `lg` (24px) top margin from the last step, `2xl` (48px) internal padding block. Primary button style = reuse the existing "Irei fazer esta receita" CTA visual treatment (`background: var(--t-bg-hero)`, `color: var(--t-hero-fg)`, `borderRadius: 18`, full-width, `padding: 17`) for "Criar minha conta". Secondary = ghost/outline variant, same shape, `border: 1px solid var(--t-bd-card)`, `color: var(--t-text-title)`, transparent background, for "Importar a minha receita".

**Responsive/mobile-first:** Identical breakpoint behavior to the existing recipe page (which has none beyond the app's fixed mobile-first single-column shell — onFeed is a mobile-first PWA-style layout with a max content width inherited from the app shell, not a responsive grid). No new breakpoints needed.

**No bottom nav / TopBar assumption:** This is the app's first logged-out surface. Do NOT render the authenticated app's `TopBar` or bottom tab bar (`--t-bg-tab-bar`) — those assume a signed-in session and app-shell chrome the visitor doesn't have context for. The public page should render as a standalone article: back button (existing `BackButton` component, `fallbackHref="/"` — safe as-is, routes an anonymous visitor to the marketing/landing `/` which itself is public) + content + conversion CTAs at the bottom. No sidebar, no nav chrome.

### 2. "Imported" ownership badge — owner's private search/swipe results (D-10)

**Placement — `ResultCard.tsx` (list view):** Add badge in the top-left position of the thumbnail, same slot pattern as the existing rank badge (`position: absolute, top: 10, left: 10`) when no rank is present, or offset below/beside it when both would collide (only owner-scoped queries surface `imported` results, and rank badges only appear in specific curated contexts — verify at implementation time whether both can co-occur; if they can, stack badge below rank badge with 6px gap, not overlapping).

**Placement — `SwipeDeck.tsx` `DeckCard` (swipe view):** Follow the exact precedent of `isVariant` → `variant-glow` class + shimmer sweep, but the imported badge is DELIBERATELY less ornamental than the variant treatment (variant = achievement/social-proof glow; imported = simple ownership label, no shimmer, no glow animation). Render as a static pill badge only — do not add a new shimmer/glow CSS class for this state.

**Visual spec (both surfaces):**
```
Background: var(--t-bg-section)   (same neutral pill background as BadgePill)
Text color: var(--t-text-secondary)
Font: 10.5-11px, weight 700
Padding: 4px 9px
Border-radius: 20px (pill)
Copy: "🎬 sua importação"
```
This deliberately mirrors `BadgePill`'s existing visual weight (occasions/equipment tags on the recipe page) — an ownership badge should read as informational metadata, not as an achievement (that's what the gold/silver/bronze rank badges and variant glow are for). Do NOT use terracota or warn colors for this badge — it is neutral, not a call to action or a warning.

**Data contract:** `SearchHit.source === "imported"` already exists in `web/lib/types.ts` (line 56) — no new field needed. Badge renders only in the owner-scoped result set (per D-10, `DEFAULTS.sources` only includes `imported` when `ownerId` is passed) so no extra "is this mine" check is needed client-side beyond checking `source === "imported"` within an already owner-scoped response.

### 3. Share affordance on owner's recipe page (SOC-02 supporting UX)

**No new component.** `ShareButton` (`web/components/ShareButton.tsx`) already exists, is already wired into `recipe/[id]/page.tsx`'s top action row, and already does exactly what's needed: native share sheet (`navigator.share`) with clipboard-copy fallback + "Copiado!" 2-second confirmation state. The ONLY change needed for this phase is what URL it shares — today it defaults to `window.location.href` (the authed `/recipe/[id]` URL); for an imported recipe still `visibility:"private"`, it must share the **`/r/[shareSlug]`** URL instead (via the existing `getUrl` prop override, already supported: `getUrl={() => \`${origin}/r/${recipe.shareSlug}\`}`). No visual change. Confirm this at planning time as a one-line prop wiring, not a new UI element.

### 4. Like button state — logged-out visitor (D-01)

**No new component.** `LikeButton` already accepts a `canLike: boolean` prop and already renders correctly for `canLike={false}`: `cursor: "default"`, `title="Faça login para curtir"`, muted (unliked, `text-carvao/40`) visual, and the `toggle()` handler already no-ops when `canLike` is false. **Gap to close:** today the false-`canLike` path is a silent no-op (tooltip only); D-01 requires it to **redirect to sign-in**. The click handler needs a new branch: `if (!canLike) { router.push(signInUrl) — no visual change to the button itself. Same disabled-looking-but-not-disabled visual (matches D-02's "visible but routes to sign-in" pattern used for account-only CTAs elsewhere on this page).

**Interaction states summary (all four surfaces already covered by existing component states — table for executor reference):**

| State | Visual | Existing component |
|-------|--------|---------------------|
| Logged-in, not liked | Outline pill, `border-areia`, muted heart | `LikeButton` (no change) |
| Logged-in, liked | Filled pill, `border-terracota/40 bg-terracota/10 text-terracota`, filled heart | `LikeButton` (no change) |
| Logged-out (any state) | Same as "not liked" visual, `cursor: default`, tooltip "Faça login para curtir" | `LikeButton` — ADD sign-in redirect on click (behavior only, zero visual change) |

## Open Questions / Assumptions (flagged for planner, not blocking)

1. **Clerk public-route exception for `/r/[token]`:** `web/middleware.ts`'s matcher currently applies `clerkMiddleware()` broadly. Confirm at plan time whether placing the route outside the `(main)` group is sufficient, or whether `clerkMiddleware` needs an explicit `isPublicRoute` allowlist entry for `/r/:token`. This is a routing/backend concern but gates whether the visual contract above can render at all for logged-out visitors — flagging so it isn't missed.
2. **Rank badge + imported badge co-occurrence in `ResultCard`/`DeckCard`:** assumed low/no collision risk since rank badges appear in curated top-N contexts and imported badges only appear in owner-scoped results — verify empirically once search widening (D-10) is implemented; stacking rule given above as a fallback.
3. **404 page for invalid/expired `/r/[token]`:** spec allows either Next's default `notFound()` (fastest, consistent with existing `recipe/[id]/page.tsx` precedent) or a lightweight branded message — no bespoke illustration/empty-state required; executor's discretion, default to reusing `notFound()` since that's the established codebase pattern (zero new UI).
4. **"Promotion state cue" pill:** CONTEXT explicitly says "keep minimal" and it is not required by any SOC/RCP success criterion — treated as optional polish. If time-constrained, planner may cut this entirely from Phase 5 scope without violating the design contract.
