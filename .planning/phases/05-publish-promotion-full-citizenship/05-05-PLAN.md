Ship the app's first logged-out surface: a public share page at `/r/[token]` that renders a confirmed import read-only with creator credits and conversion CTAs, lets an anonymous visitor like it (routing to sign-in), and wires the owner's share affordance to the token URL — all reusing the existing recipe page's visual system with zero new design.

This plan adds the Next public route (outside the Clerk-protected (main) group), the middleware public-route allowlist, the LikeButton sign-in redirect, the ShareButton token wiring, and the web api/type plumbing.
Output: web/lib types+api for share fetch, the middleware allowlist, the /r/[token] page (reusing recipe/[id] layout + sourceMeta credits + conversion CTAs), the LikeButton redirect, and the owner-page ShareButton wiring.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| anonymous browser -> /r/[token] page | The first logged-out UI; it must render only what a token-holder should see and route all account actions to sign-in |
| middleware -> route protection | The public-route allowlist decides which paths bypass Clerk; an over-broad matcher would expose authed routes |
| logged-out like -> sign-in | An anonymous like must convert to a real authenticated like (1 account = 1 like) to keep the promotion gate honest (D-01/D-08) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-15 | Elevation of Privilege (auth bypass via matcher) | web/middleware.ts public route | high | mitigate | createRouteMatcher scoped to exactly `/r/(.*)`; auth.protect() on all other routes; the no-key fallback preserved — no authed route is exposed by the allowlist |
| T-05-16 | Spoofing (anonymous like fraud) | LikeButton logged-out path | high | mitigate | Logged-out like redirects to sign-in instead of recording an anonymous like — every like ties to a real account (D-01), keeping the +5 promotion gate un-inflatable by anonymous traffic |
| T-05-17 | Information Disclosure (private data on public page) | /r/[token] page render | medium | mitigate | The page renders only the recipe returned by the token-gated backend (Plan 03); no objectId fetch, no owner-only fields; the source video is linked (sourceMeta) never re-hosted (SOC-03/PROJECT decision) |
| T-05-18 | Tampering (token URL leakage to wrong surface) | ShareButton getUrl | low | mitigate | Owner ShareButton shares the token URL only for private imports; public recipes share the canonical URL — no accidental token exposure beyond the intended share |
| T-05-SC | Tampering | npm/pip/cargo installs | high | mitigate | No new packages (Clerk already installed; createRouteMatcher is from the existing @clerk/nextjs). If any install is proposed, gate via slopcheck + blocking human checkpoint for [ASSUMED]/[SUS] |
